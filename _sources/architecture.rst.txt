=================================
Architecture (Multi-Agent System)
=================================

The core innovation of the Tensorplane AI Foundry is its transition from static infrastructure management to **Scalable Agency**. Instead of relying on human operators to profile and patch I/O bottlenecks, Tensorplane utilizes a specialized Multi-Agent System (MAS) to autonomously co-design and compile hardware-specific data paths.

The Model Context Protocol (MCP)
================================

To achieve true recursive self-improvement, the system leverages the **Model Context Protocol (MCP)**. MCP provides a standardized, stateless communication layer that allows our agentic pools to read real-time telemetry, access the global infrastructure state, and execute hyper-specialized tools securely.

- **Distributed FaaS Deployment:** MCP servers are deployed statelessly via AWS Lambda.
- **S3 Context Hydration:** Tools dynamically fetch their context from S3 URIs, allowing agents to share massive trace logs and eBPF maps without local file system constraints.

The "Team of Agents" Topology
=============================

The MAS is strictly hierarchical, organized into functional pools under the ``/team_of_agents/`` directory.

1. The Orchestrator (AgentOps)
------------------------------
Acting as the central "brain," the Orchestrator receives high-level intents (e.g., "Optimize the Ray cluster for Llama-3 70B inference"). It decomposes these prompts using a **Stage Generation Agent**, maps the stages to specific MCP tools via a **Planner Agent**, and delegates the workload to the specialized pools below.

2. Ops: Telemetry & Triage
--------------------------
This pool acts as the autonomous SRE team.
* **Performance Monitor:** Hooks into Cilium/Hubble eBPF maps to track P99 tail latency and identify POSIX socket overhead.
* **Resource Optimizer:** Manages GPU fractioning, automatically adjusting NVIDIA Multi-Instance GPU (MIG) slices based on the incoming inference queue.
* **Power Monitor:** Reads hardware Performance Monitoring Units (PMUs) to manage thermal limits and Dynamic Voltage and Frequency Scaling (DVFS).

3. Dev: The Builders
--------------------
When the Ops pool detects a bottleneck, the Dev pool is dispatched to rewrite the underlying infrastructure code.
* **C++ Data Plane Agents:** Specialized in writing lock-free ``io_uring`` and SPDK code.
* **Rust Kernel Agents:** Generate eXpress Resubmission Path (XRP) eBPF hooks to push index lookups down into the NVMe driver.
* **Bare-Metal Agents:** Write inline RISC-V assembly tuned for Stream Semantic Registers (SSR) on the Snitch architecture.

4. Eval & Deployment
--------------------
Code generated by the Dev pool is never deployed blindly. The Eval agents compile the code in isolated sandboxes, run cycle-accurate benchmarks, and perform hot-swapping into the live data plane only if the new kernel mathematically proves to lower latency or increase throughput.

The Recursive Loop
==================

1. **Observe:** ``Ops`` detects an I/O stall during multi-node tensor reduction.
2. **Plan:** ``Orchestrator`` dispatches a task to rewrite the NVMe polling logic.
3. **Build:** ``Dev (Rust)`` generates a new ``usrbio`` lock-free queue implementation.
4. **Evaluate:** ``Eval`` compiles the crate, runs a microbenchmark, and validates a 15% throughput increase.
5. **Deploy:** The new binary is injected into the data plane, and the cycle repeats.
